Handler总结

Handler总结

1. handler的本质

1. 用于线程通信。
2. handler sendMessage 发送到消息队列，然后loop取出进行分发，最后到handler.handleMessage()里面
3. looper.loop起始是 message.target.dispatch()

2. 一个线程有几个Looper？有几个messageQueue？如何保证？

1. 本质通过ThreadLocal
2. Looper.prepare的时候检测，通过threadlocal检测，如果 已经有，直接跑异常。
3. 保证一个线程只有一个ThreadLocal。
4. MessageQueue是Looper的一个成员变量。

3. 一个线程可以有几个handler？那如何保证message能分配给正确的handler 处理呢

1. Activity可以初始化，service也可以初始化，因此一个线程可以有多个handler。
2. handler.sendMessage, 这个时候设置了message.target = this(handler);
3. loop的时候最终调用了 msg.target.dispatch()方法。

4. message Queue的数据结构

1. 单向链表
2. 按照时间进行排序的
3. 优先取头部数据

5. 主线程做了哪些事情，可以直接 new Handler

1. prepareMainLooper()
2. loop方法，开启死循环

6. postDelay时间准确吗？

1. 不完全准确
2. 消息是一个一个处理的，如果一个耗时，另外一个就不准。

7. Handler为什么不会ANR

1. 底层的阻塞机制，
2. ANR是消息没有及时处理。

8. IdleHandler的使用

1. looper.loop最终是调用了 messagaQueue.next()方法
2. next会先检测是否有message，没有的话或者未到执行时间会执行idleHandler的queueIdle方法

9. 同步屏障是什么？

1. handler有同步消息，异步消息，屏障消息。
2. 屏障消息的核心是 msg.target == null
3. 正常情况下，同步消息、异步消息是一样的；遇到同步屏障消息的时候，优先执行异步消息。
4. 绘制任务是一个异步消息，也是通过同步屏障能够保证优先执行

10. handlerThread

1. 本身是一个THread
2. start()的时候初始化looper,开启loop循环。handleMessage的线程核心是looper的线程
3. 一般用于需要顺序执行的异步操作。

11. IntentService使用

1. 继承service
2. onCreate里面使用了HandlerThread
3. handleMessage里面，重写了onHandleIntent()
4. 自动停止。

12. handler内存泄漏

1. 匿名内部类持有外部类引用
2. handler持有activity引用
3. message持有handler引用
4. messagequeu持有message引用
5. looper持有messagequeue引用
6. looper是线程私有的
。。
所以有延迟消息的时候，会导致activity无法被回收，因此需要移除消息。
